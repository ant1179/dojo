# Workflow Template

> This template defines the standard structure for every workflow used by the AI Engineering Team.
>
> A workflow describes **how the team collaborates**, not how code is written.

---

# Metadata

```yaml
name:
description:
version: 1.0
owner: Technical Lead
```

---

# Purpose

Why does this workflow exist?

Describe the business objective.

Example:

Deliver a new feature from specification to production-ready Pull Request.

---

# Trigger

When should this workflow start?

Examples:

- New feature approved
- Bug reported
- Release candidate ready

---

# Participants

List every role involved.

Example:

- Product Owner
- Technical Lead
- Product Spec Review
- Architect Review
- Backend Feature
- Frontend Screen
- QA Review

---

# Workflow

For every phase define:

## Phase Name

### Responsible

Who owns this phase?

### Input

What information is required?

### Activities

Describe what happens.

### Deliverables

What must be produced?

### Exit Criteria

What must be true before continuing?

### Next Phase

Who receives the work?

---

# Escalation Rules

Describe when the workflow must stop.

Examples:

- Missing requirements
- Architecture conflict
- Security issue
- Failed Quality Gates
- Blocking dependency

---

# Artifacts Produced

List every artifact generated.

Examples:

- Specification
- Architecture Review
- Pull Request
- Test Report
- QA Report

---

# Success Criteria

The workflow is complete only if:

- All phases completed
- Every Quality Gate passed
- Documentation updated
- Human Product Owner approved

---

# Failure Criteria

The workflow fails if:

- Work bypasses a phase
- Required deliverables are missing
- Quality Gates fail
- Human approval is skipped

---

# Continuous Improvement

After completion answer:

- What slowed the workflow?
- Which steps could be automated?
- Which skills collaborated well?
- Which documentation was missing?

Document improvement opportunities.
